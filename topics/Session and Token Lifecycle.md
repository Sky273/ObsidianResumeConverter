# Session and Token Lifecycle

## Summary

ResumeConverter uses a cookie-oriented auth model with separate access and refresh tokens, plus CSRF on authenticated state-changing flows.

The important mental model is:

1. sign in sets cookies
2. protected requests use the `accessToken` cookie
3. refresh rotates both tokens
4. CSRF token fetch is a separate step for state-changing requests
5. logout revokes both tokens and clears both cookies

## Main Cookies

### Access token

- cookie name: `accessToken`
- path: `/`
- `httpOnly: true`
- `sameSite: 'lax'`
- max age: 1 hour
- `secure` depends on request security and deployment mode

This cookie is sent broadly across the app and powers authenticated API access.

### Refresh token

- cookie name: `refreshToken`
- path: `/api/auth`
- `httpOnly: true`
- `sameSite: 'lax'`
- max age: 7 days
- `secure` depends on request security and deployment mode

This cookie is intentionally narrower in scope than the access token.

### CSRF token

- cookie name: `x-csrf-token`
- generated by `GET /api/csrf-token`
- paired with the `x-csrf-token` request header in the double-submit model

This cookie is not the authentication token; it exists to protect cookie-authenticated mutations.

## Sign-In Flow

Primary route:

- `POST /api/auth/signin`

Flow:

1. request passes auth rate limiting and body validation
2. backend validates email/password format
3. backend loads the user with firm data
4. backend blocks bad states:
   - unknown user
   - invalid password
   - inactive account
   - unverified self-service user
   - missing firm assignment
   - password replacement required
5. if 2FA is enabled, a valid TOTP or backup code is required
6. backend updates `last_login`
7. backend issues a new `accessToken` and `refreshToken`
8. cookies are set and the user payload is returned

Security consequence:

- sign-in is not only credential verification; it is also an account-state gate

## Authenticated Request Flow

Protected routes rely on backend middleware rather than frontend guards alone.

Current request behavior:

1. backend reads `req.cookies.accessToken`
2. token is verified
3. current user is reloaded
4. `inactive` users are rejected
5. token lifetime headers are emitted:
   - `X-Token-Expires-In`
   - `X-Token-Expiring-Soon` when near expiry

There is also a very short-lived authenticated-user cache to avoid repeated DB lookups on tightly grouped requests.

## Current User Revalidation

Route:

- `GET /api/auth/me`

Behavior:

- requires valid authenticated access token
- returns fresh user-with-firm data
- blocks users with missing firm assignment
- sends revalidation-friendly cache headers

This route is the authoritative way for the SPA to recover current authenticated user state.

## Refresh Flow

Primary route:

- `POST /api/auth/refresh`

Behavior:

1. request uses the `refreshToken` cookie
2. backend consumes the refresh token instead of merely checking it
3. current user is reloaded
4. invalid or inactive users are rejected
5. missing firm assignment is blocked
6. backend issues both a new access token and a new refresh token
7. both cookies are overwritten

Important consequence:

- refresh is token rotation, not static token reuse
- stolen old refresh tokens are less useful after they are consumed

## Logout Flow

Primary routes:

- `POST /api/auth/logout`
- `POST /api/auth/signout`

Behavior:

1. backend tries to resolve the user from access token first, then refresh token
2. backend revokes both tokens if present
3. backend logs a logout security event
4. backend clears both cookies with the correct path-aware clear options

This means logout tries to invalidate both the browser session and the server-side token trust.

## CSRF Lifecycle

CSRF is separate from sign-in itself.

Current model:

1. browser fetches `GET /api/csrf-token`
2. backend generates a CSRF token and cookie
3. frontend includes the `x-csrf-token` header on state-changing authenticated requests
4. backend validates cookie/header pairing via double-submit protection

Important exemptions currently exist for:

- `/api/auth/signin`
- `/api/auth/refresh`
- `/api/auth/logout`
- `/api/auth/register`
- `/api/consent/respond/*`
- `/api/gdpr/mail/callback`

Operational consequence:

- the app can support first-load login/refresh flows without already having a CSRF token
- after auth bootstrap, mutations still rely on explicit CSRF token handling

## Secure Cookie Behavior

Cookie `secure` behavior is not hardcoded blindly.

The auth cookie config checks:

- `NODE_ENV`
- `HTTPS_ENABLED`
- `req.secure`
- `x-forwarded-proto`

This is important behind proxies and reverse proxies. If the deployment is meant to be secure but the request is not seen as HTTPS, cookies may be emitted without `Secure` and the backend logs that fallback.

## Typical Failure Modes

### User appears logged out after page reload

First suspects:

- expired access token with broken refresh flow
- missing refresh cookie on `/api/auth`
- reverse-proxy cookie/path issue

### User can sign in but mutations fail

First suspects:

- missing CSRF bootstrap call
- stale/invalid `x-csrf-token` cookie/header pair

### Logout appears partial

First suspects:

- cookies not cleared on matching paths
- revoked tokens not aligned with browser cookie state

### Session behaves differently in Docker/proxy

First suspects:

- wrong forwarded-proto handling
- incorrect `Secure` cookie emission expectations
